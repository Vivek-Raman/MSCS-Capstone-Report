\subsection{Build pipeline}

The build pipeline consists of several stages.

\begin{enumerate}
\item First, the sidecar is compiled and packaged into a standalone binary using the PyInstaller system.
\item The Tectonic binary for LaTeX compilation is then downloaded from its repository.
\item Both binaries are copied into the Tauri source code and registered for bundling.
\item The Tauri frontend is compiled. This includes the other binaries and creates an installable package.
\item These steps are repeated on all supported operating systems.
\item The generated installers for each OS are uploaded and linked to the code repository.
\item In parallel, the server is built and packaged as a Docker image, ready to be pulled and deployed to a server.
\end{enumerate}

\subsection{Agent design}

The agent is implemented as a simple LangGraph state machine with a deterministic execution graph \cite{langgraph}.
The agent adheres to a system prompt that defines its role and capabilities.
It is tasked with interpreting user requests and making intelligent document modifications.
The prompt also establishes guidelines for the editing workflow (list files $\rightarrow$ read $\rightarrow$ edit $\rightarrow$ compile).
The safety mechanisms are also defined in the prompt.
The complete system prompt is available in Appendix \ref{apx:system-prompt}.

\subsubsection{Tool invocation}

When the agent decides to take an action, it invokes the appropriate tool.
The tool result is then fed back into the agent.
This forms a loop that allows the agent to respond to outcomes and handle errors.
In case the agents needs additional clarification, it can request the user for information.

A typical user request is handled as follows.

\begin{enumerate}
\item The agent reads the user request and starts by using the list\_files\_tool to understand the directory structure.
\item Then it reads main.tex to understand the layout of each section.
\item Based on the user's request, the agent reads the specific sections needed to address the request.
\item If there is an image, the agent uses the appropriate tool to import the image into the project.
\item The agent writes to the requested files.
\item The compilation tool is then invoked to check for errors.
In case of any errors, the agent repeats the process to resolve them.
\item Finally, the agent responds with a summary of its actions.
\end{enumerate}

\subsubsection{Safety \& ethical constraints}

The agent is designed with built-in safety mechanisms.

\begin{itemize}
\item All file operations validate paths at the tooling level to prevent escaping the project directory.
\item The prompt instructs the agent not to add undesired content.
\item Compilation errors trigger errors that demand the agent's attention.
\end{itemize}

These constraints guarantee that the agent acts as a trusted and consistent scribe rather than an autonomous editor.

\subsection{\LaTeX\ compilation}

The \LaTeX\ compilation subsystem is linked to the frontend, similar to the sidecar server \cite{tauri_sidecar}.
The system uses the Tectonic typesetting engine as its core compilation backend \cite{tectonic}.

Compilation can be initiated by the user or by the agent using the compile tool.
When it is initiated, the sidecar invokes the Tectonic engine to compile the document.
This created the PDF file in the project directory.
In case of errors, the engine logs the error to standard error (stderr).
This is captured and forwarded to the user or the agent for debugging.

\subsection{Benchmark}

The benchmark module is designed to test the performance of various language models using the Spartan Write agent.
This is accomplished using a reproducible test suite comprising a series of test documents and user requests.
The benchmark module consists of four core components:
\begin{itemize}
\item a CLI runner for orchestration;
\item a scoring engine for metric evaluation;
\item SQLite database storage for persistence;
\item and a Streamlit dashboard for result visualisation.
\end{itemize}

The benchmark execution follows a well-defined pipeline.
\begin{itemize}
\item First, the dataset is loaded into the result directory and the database is initialised.
\item For each test case, the tool prepares a workspace directory and sends the prompt to the agent server.
\item After the chat completes, the scoring engine processes results to extract metrics.
\item These metrics are stored in the database.
\item The dashboard then aggregates results from the database and renders interactive visualisations.
\end{itemize}

\subsubsection{Tests design}

The tests written as part of the benchmark are tabulated in Table \ref{tab:tests}.
They can be categorised as follows.

\begin{tabularx}{\textwidth}{|l|X|}
\hline
\textbf{Test ID} & \textbf{Task for the agent} \\
\hline

\hline
001-title-and-sections & Set document title and scaffold initial sections. \\
\hline
002-add-contents-to-section & Add multi-line content to an existing section. \\
\hline
003-remove-contents-from-section & Remove a specific line from a section. \\
\hline
004-large-unarranged-doc & Restructure and fix an unorganised document's file structure. \\
\hline
005-summarise-large-doc & Generate a concise summary of a large document. \\
\hline
006-large-new-section & Add a new section with specific detailed contents. \\
\hline
007-large-remove-contents & Remove a specific multi-line passage from a document. \\
\hline
008-large-search-fuzzy & Locate and pinpoint all discussions of a semantic topic. \\
\hline
009-large-remove-fuzzy & Remove all content discussing a particular semantic topic. \\
\hline
010-large-rearrange-sections & Swap the positions of two major sections. \\
\hline
011-large-rearrange-subsections & Swap subsections within a section. \\
\hline
012-fail-to-generate-content & Attempt to generate content to test the agent's resilience. \\
\hline
013-large-add-image & Insert an attached diagram at a specified location. \\
\hline
014-large-broken-explicit-fix & Repair a broken LaTeX document using explicit instructions. \\
\hline
015-large-broken-implicit-fix & Repair a broken document through implicit problem inference. \\
\hline
016-large-parse-image & Interpret and describe the contents of an attached image. \\
\hline
\end{tabularx}
\caption{Benchmark test cases}
\label{tab:tests}
\end{table}

\begin{enumerate}

\item \textbf{Document structure manipulation} \\

Tests 000, 001, 002, 003, 004, 006, 010, and 011 validate the core ability to create and modify the structure of the document.
Each test assesses whether the LLM can correctly parse and modify the document hierarchy.
Failures in this category would indicate difficulty understanding parent-child relationships in \LaTeX.
These tests verify the basic functionality expected of the agent.

\item \textbf{Search and targeted deletion} \\

Tests 007 to 009 assess how well the language model can find and remove content from long documents.
These tests look at how accurately the model can follow instructions when asked to delete a specific item
They also test how well it can make sense of less direct requests, like removing all mentions of a certain topic.
If a model fails here, it usually means it has trouble recognising context or isn't precise enough when making changes.

\item \textbf{Document contents} \\

Tests 005 and 012 check how well the language model works with the document contents.
Test 005 is all about distilling a large document into a clear, readable summary.
Test 012, on the other hand, asks the model to generate new material that adheres to the style and scope of what's already there.
Test 012 must be denied by the agent, as generating new content is seen as a violation of integrity.

\item \textbf{Error recovery and problem diagnosis} \\

Tests 014 and 015 look at how well the model can spot and fix broken \LaTeX\ documents.
Test 014 is simple: it tells the model plainly, "Fix the document."
Test 015 expects the model to realise on its own that something needs repairing.
These tests show whether the model can diagnose problems and respond to both clear and subtle hints.

\item \textbf{Multimodal compatibility} \\

Tests 013 and 016 see if the model can handle images as part of the request.
Test 013 checks if the model can insert an image where it belongs, and Test 016 asks for a description of what's shown in the picture.
If the model has trouble here, it might not understand images, generate the right file references, or know where to place things in the document.

\end{enumerate}

\subsubsection{Scoring metrics}

The scoring engine evaluates three primary metrics across each benchmark run.
\textbf{Tool use} aggregates tool calls by name and counts invocations per tool type.
This provides insight into which capabilities the LLM uses.
\textbf{Chat duration} calculates the time taken for the agent to complete the task.
This is used to identify degradation in performance due to LLM updates or provider outages.
\textbf{Token usage} sums input tokens, output tokens, and reasoning tokens across all AI messages.
This tracks resource utilisation.
