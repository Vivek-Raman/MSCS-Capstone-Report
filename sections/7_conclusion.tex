\section{Conclusion}

This report introduces Spartan Write, a document authoring software powered by agentic AI.
It discusses the shortcomings of authoring documents with general purpose word processing software.
Then, it highlights the steep learning curve that \LaTeX-based solutions like Overleaf offer.

\subsection{Key findings and interpretation}
The benchmark turns up a somewhat surprising insight.
Differences in cost and latency among modern LLMs are due to the way they use tools.

All capable models complete tasks at roughly the same rate (Table~\ref{tab:task-completion}).
However, they get there using very different strategies.
For example, DeepSeek repeatedly reads files it doesn't need, which pushes its token use up to $62{,}485$ per run, and results in a mean duration of $148.6$s (Figure~\ref{fig:bench-duration-vs-tokens}).
This is not a problem with DeepSeek's reasoning.
It is a side effect of tool usage behaviour.
Gemini~3~Flash restructures in place using \texttt{rename\_file} (Figure~\ref{fig:bench-tool-use}).
This delivers the lowest latency ($12.9$s) and the second lowest cost ($\$0.0094$ per run, Figure~\ref{fig:bench-cost-per-model}).

For document-editing workloads, raw model ability is only part of the story.
The better predictor is whether the model settles into efficient tool-use patterns.
That also means prompt and tool-schema design matter.
A model can look expensive simply because the interface pushes it toward wasteful actions.
The GPT family, for instance, never uses \texttt{rename\_file} in any completed run, even when a rename-based solution is available (Figure~\ref{fig:bench-tool-use}).
That looks less like a capability gap and more like a mismatch between the model and the schema.

Across all models, the hardest tasks involve large structural changes (Table~\ref{tab:hardest-tasks}, Figure~\ref{fig:bench-duration-by-job}), not weaknesses tied to any specific model.
This is a signal that the true complexity of document editing comes from the nature of the work itself.

Based on these findings, \texttt{google/gemini-3-flash-preview} stands out as the recommended primary model for Spartan Write.
It is fast and cheap, and aligns well with document editing workflows.
That said, this recommendation is based on efficiency rather than correctness.
Completing a task quickly and cheaply does not always mean the result is the best possible.

\subsection{Contributions}

This work makes two contributions to the LLM-driven document authoring space.

First, the report presents Spartan Write, a document authoring system demonstrating how agentic AI reduces cognitive load in \LaTeX-based technical writing.
Rather than requiring users to master the syntax for editing, the system abstracts file operations and document structure behind a natural language interface.
This lets users express intent and delegate implementation details to the agent.

Second, it contributes a benchmark methodology for evaluating LLMs on document editing tasks.
The suite comprises 17 test cases spanning various use cases for the agent like editing operations, large refactors, error correction, and multimodal tasks.
It includes metrics for reliability, latency, token usage, cost, and tool invocation patterns.

\subsection{Limitations and future work}

The key limitation is that this benchmark measures \textbf{efficiency} rather than output \textbf{correctness}.
The bench reports task completion rates and other performance metrics, but \textit{completion} here only means the agent completed the task.
It doesn't guarantee that edits will always match what the user wanted.
So, the recommendation for Gemini is based solely on efficiency, and it assumes that all models deliver equally correct results.

If output quality varies significantly across models, the efficiency-based rankings could be misleading.
In practice, a model that is a bit slower but more reliable might be a better choice, even if the benchmark suggests otherwise.
A more complete evaluation of output quality would help fill this gap by specifying specific operations on a case-by-case basis.
For example, the content generation test could include an assertion that the write tool is never called.
Other possible approaches include comparing edits to reference versions, structured annotation, or asking an LLM to judge a sample of benchmark runs.
Adding these quality metrics is the main priority for future work, and would give more useful context for choosing a model.

Beyond quality assessment, two secondary research directions remain.
The first is investigating the GPT family's systematic avoidance of \texttt{rename\_file}.
The second is validating the benchmark against real-world usage through human-centered studies measuring qualitative differences through user feedback.

Alongside assessment improvements, several Spartan Write UX feature enhancements merit exploration.
These include undo support for agent actions, real-time multi-user collaboration, and integration of SyncTeX, which would allow the agent to edit documents more accurately by mapping PDF coordinates back to source locations.
