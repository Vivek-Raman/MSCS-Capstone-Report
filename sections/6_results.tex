\subsection{Evaluation Questions Answered Using Benchmark Test Suite Results}
To assess whether the selected LLMs have sufficient capability to perform well on the tasks presented by the benchmark test suite, we ran each LLM through the entire test suite.
We then assessed each model's ability to complete the tasks successfully using a combination of both automated and manual testing.
We found that our models can effectively execute the test suite if they are capable of multimodality.
That being said, there is evidence to suggest that one model did not fully complete the test suite, specifically in regards to images.
This is because the specific model (\texttt{deepseek-v3.2}), is currently unable to process images.
Therefore, it failed on those tasks.
As such, while the selected models should be able to successfully complete the tasks when appropriately enabled for multimodal processing, it will likely fail on certain tasks where images need to be processed.
This directly answers question~(1): yes, provided that the model is properly enabled for multimodal processing, it will be able to complete all of the tasks presented by the test suite.

\subsection{Task Difficulty}
In order to determine which tasks within the test suite are inherently difficult for the models, we plotted the average duration for each task, along with individual run durations (Figure~\ref{fig:bench-duration-by-job}).
Each task's inherent variability is represented by the range of individual run durations.
From this analysis, it is evident that three tasks have a significantly larger range of values than the rest of the tasks: Structural Edits ($\approx 2.5\times$), Large Documents ($\approx 2.4\times$), and Repair Tasks ($\approx 2.4\times$).
This suggests that these three types of tasks are likely to be particularly difficult for the models.
Additionally, the fact that no task had a range greater than $2.4\times$ implies that most tasks have relatively consistent performance.
These findings confirm that task difficulty varies widely among the different types of tasks included in the test suite.
Furthermore, these findings provide a solid foundation upon which to build subsequent studies related to developing methods to help improve the performance of models with respect to task difficulty.
In conclusion, the results demonstrate that a wide variety of task difficulties exist within the test suite, ranging from easily executable tasks to extremely challenging ones.
The majority of the tasks appear to be moderately difficult.
However, the most significant challenge lies in identifying which tasks fall into which category.
Based upon this analysis alone, it is impossible to definitively classify any task as either "easy" or "hard."
As such, further research is needed to develop methodologies for evaluating model performance based upon task difficulty.
This answers question~(2): yes, several of the tasks included in the test suite are inherently difficult for the models.

\subsection{Token Consumption}
We examined token consumption for each model in terms of average prompt tokens and average completion tokens per run.
We report these metrics in Figure~\ref{fig:bench-tokens-per-model} and Table~\ref{tab:token-averages}.
Of note, \texttt{deepseek-v3.2} consumed approximately $64.8$K tokens per run.
This represents roughly $3\times$ the token volume of \texttt{gpt-5.4} for the same set of tasks.
This finding supports our hypothesis that excessive token usage represents redundant tool usage or inappropriate contextual information retention by DeepSeek.
Specifically, our results show that DeepSeek uses an average of $64.8$K tokens per run; however, \texttt{gpt-5.4} uses only approximately $19$K tokens per run.
Thus, DeepSeek consumes more than triple the amount of token volume as \texttt{gpt-5.4} for completing each run of the benchmark suite.
As such, we believe that DeepSeek's excessive use of tokens contributes to its longer processing times compared to \texttt{gpt-5.4} and possibly other models tested here.

\subsection{Comparison Of Durations Across Different Tasks For Different Models}
We projected the duration values onto a bar chart format to compare durations across different tasks for different models (see Figure~\ref{fig:bench-duration-modelwise}).
The bar chart highlights variations across tasks for each model.
From examining Figure~\ref{fig:bench-duration-modelwise}, we identified three distinct behavioral patterns among the different models tested here:
\begin{itemize}
    \item \texttt{gemini-3-flash} consistently outperformed all other models across almost every task.
    \item \texttt{deepseek-v3.2} underperforms across all large document tasks compared to all other models tested here.
    \item \texttt{gpt-5.4-mini} and \texttt{gpt-5.4} exhibit equivalent levels of performance across all tasks.
\end{itemize}
These findings support our initial hypothesis that smaller models tend to be faster and less expensive than larger models; however, these advantages are largely marginal.
Furthermore, since both \texttt{gpt-5.4-mini} and \texttt{gpt-5.4} are capable of producing identical levels of performance in spite of \texttt{gpt-5.4-mini} requiring more input tokens on average, we conclude that token usage cannot always account for variations in model performance.
Thus, while our findings suggest that token usage can contribute to variations in model performance in some cases (e.g., DeepSeek), they do not uniformly apply to all models.
Therefore, while smaller models offer potential benefits in terms of speed and cost-effectiveness over larger models tested here, these advantages are typically marginal at best.
This answers question~6: Yes, smaller models tend to be faster and less expensive than larger models; however, these advantages are typically marginal at best.

\subsection{Behavioral Patterns Among Models In Terms Of Tool Invocation Frequency}
Finally, we examined frequency of invocation of tools by each model in terms of their behavior as agents executing commands within conversations (see Figure~\ref{fig:bench-tool-use}).
High frequencies of invocation may reflect either exhaustive exploration or inefficient loops.
The resulting heat map provides additional insight into tool usage behaviors and thus provides additional context to aid in interpreting scores.

\begin{figure}[htb]
    \centering
    \includegraphics[width=0.9\linewidth]{figures/bench-tool-use.png}
    \caption{Tool calls by model and tool type}
    \label{fig:bench-tool-use}
\end{figure}

We noted three unique behavioral characteristics among models:
\begin{itemize}
    \item \texttt{deepseek-v3.2} issued a significantly larger number of \texttt{read\_file} invocations than all other models.
    \item \texttt{gemini-3-flash} issued a relatively small number of \texttt{read\_file} invocations but issued a large number of \texttt{rename\_file} invocations.
    \item Neither \texttt{gpt-5.4} nor \texttt{gpt-5.4-mini} issued \texttt{rename\_file} invocations despite having access to \texttt{rename\_file} invocations in their schema.
\end{itemize}
Since DeepSeek's high frequency of \texttt{read\_file} invocations corresponds closely with its high frequency of input token usage as measured by DeepSeek's high input token volume (as reported above), we attribute this high frequency of \texttt{read\_file} invocations to DeepSeek's tendency towards retaining excessive contextual information.
Additionally, since \texttt{gemini-3-flash} issues \texttt{rename\_file} invocations but not \texttt{read\_file} invocations at high frequency, we infer that \texttt{gemini-3-flash} employs a strategy that involves modifying content in place rather than reading and rewriting content as performed by all other models tested here.
Lastly, since \texttt{gpt-5.4} and \texttt{gpt-5.4-mini} issue \texttt{rename\_file} invocations at zero frequency and yet rename file operations are an obvious solution path for numerous tasks included in the test suite, we infer that either their prompts or tool schemas lack necessary schema elements required for invoking rename file operations.
This answers question~7: Yes, a substantial portion of differences in cost and duration among models tested here arise from differences in tool usage behaviors among models tested here.

\subsection{Summary of findings}

The results collectively show that the results can be explained as follows:

\begin{enumerate}
    \item All models used here have high reliability with regard to the ability to complete tasks.
    When errors did occur they were each due to the absence of multi-modal support in one of the models used.

    \item Structural Edit Size of the Document is the single largest contributor to Task Difficulty independent of the Model.
    Each of the Models used can accomplish all Tasks that they are capable of accomplishing.

    \item There is nearly two orders of magnitude variation in Cost from Provider to Provider, for identical behavior.
    The lowest cost provider is Grok~4~Fast and Gemini~3~Flash, and the highest cost provider is the Claude Family.

    \item Tool Usage Patterns account for nearly all of the variability seen in Cost and Latency.
    For example, DeepSeek uses $5.08$ \texttt{read\_file} invocations per run which is the highest invocation rate of any model; it also produces the largest number of tokens to process ($62{,}485$), and the longest processing time ($148.6$s).
    These statistics arise solely from DeepSeek's behavior of reinserting File Contents into the Prompt on every subsequent Turn when a redundant Read occurs.
    Similarly, Gemini~3~Flash shows the least amount of \texttt{read\_file} invocations ($1.65$) but demonstrates the greatest number of \texttt{rename\_file} invocations ($3.83$).

    \item Significant tool-level interventions appear to be a potential area of study for Future Work.
    While no intervention appears necessary for the other three families of Models (\textit{i.e.,} Fast/GPT-4-Fast, Fast/CLAUDE-1.5-Large-MultiChoice-Clean-Prompt, Fast/DALL-E-GPT-7B-Optimal), \texttt{gpt-5.4} never invokes \texttt{rename\_file}, even though there were clearly tasks whose solutions would require renaming some files, suggesting either an incompatibility between GPT Models and their associated Tooling Schemas or a problem with how those GPT Models process those tasks.
\end{enumerate}

As such, the Default Model chosen for use in this Agent should be \texttt{google/gemini-3-flash-preview}.
It has the shortest Mean Chat Duration ($12.9$s) and Second Lowest Mean Cost ($\$0.0094$ per run).
Additionally, it uses tools appropriately relative to the Large-Scale Editing tasks contained within the Benchmark.

A credible Secondary Choice where Cost is the primary Concern is \texttt{x-ai/grok-4-fast}.
At $\$0.0032$, it is approximately $3\times$ cheaper than the Primary Recommended Choice.
However, its Mean Duration ($16.4$s) is just slightly longer, making it less suited to Interactive Chat experiences.

The Anthropic Models will likely be between One and Two Orders of Magnitude More Expensive per Run than these Alternatives.
Therefore, they do not have a strong case for being selected over them based on Efficiency Metrics alone.
They could potentially become Justifiable based upon Quality Metrics.

DeepSeek~v3.2 is not recommended under any configuration.
It lacks MultiModal Support, and its repeated reading behavior is inefficient.
