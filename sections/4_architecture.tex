\subsection{Frontend}
Spartan Write's frontend is built using React.js \cite{react}, and runs using Tauri, a framework for building cross-platform native applications for Windows, macOS and Linux from a single TypeScript/React codebase \cite{tauri}.
Accessibility is a important focus on Spartan Write, we have configurable text zoom and magnifier, support for screen readers, and all UI is built with industry standard components that this taken into consideration.

\subsubsection{Development tooling and libraries}
\begin{itemize}
\item Tauri for being able use the native system api, without adding too big of a file size to our applications \cite{tauri}, and not having the need to deploy to every OS separately with their own codebase.
\item Vite for fast build times, and hot module reloading when developing \cite{vite}.
\item TypeScript for better safe typing, and used to be industry standard by now \cite{typescript}.
\item Styling is handled by Tailwind CSS for responsive, utility-first design \cite{tailwind}.
\item User interface components are derived from shadcn, a UI component library built with accessibility and consistent visual design in mind \cite{shadcn}.
\item The source code editor implements the Monaco Editor which provides syntax highlights \cite{monaco}.
\end{itemize}

\subsubsection{Authentication}
The frontend integrates WorkOS AuthKit to provide secure, standards-based authentication \cite{workos}.
The authentication framework follows best practices for security and developer and user convenience.
Once authenticated, the frontend manages user credentials and conveys usage data.
Only authenticated users are permitted to invoke the agent.

\subsubsection{Project management}
The frontend provides a dashboard to view and open recent projects.
There is an onboarding flow for new users and a settings page that shows user profile info and usage stats.

\subsubsection{Editor}
The editor has a resizable layout, containing two modes of operation.
The first is source view, a simple \LaTeX\ code editor.
The second, and default mode, is preview view, showing a live PDF render of the document.
The chat interface is always available in both, so the user can see the work that the agent is doing as it happens.

\subsection{Sidecar}
The sidecar is a FastAPI server \cite{fastapi} running locally, bundled with the desktop app.
As with other local components, its role is to link the frontend UI with the underlying project filesystem.
Unlike all the other “servers”, this program runs on the users' own machines alongside the actual client application \cite{tauri_sidecar}.
Its main responsibility is to perform all filesystem I/O, so that the cloud server can stay stateless, implementing only AI agent logic.
Among other things, the sidecar is responsible for storing project-specific, and user-level configuration: user info, API credentials (in case the user is able to provide their own API key), and authentications tokens.
The sidecar persists these settings, for fast access and offline operation.
It is responsible for all filesystem-bound operations, as well as compilation and configuration for the user.

\subsubsection{Template discovery}
This sidecar offers an endpoint to query all of the available project templates, which the frontend can display to the user when they're creating a new project.
This allows, for example, easy extensibility: a new template can be added there, without needing to change the frontend.

\subsubsection{Compilation and PDF rendering}
An important capability for the sidecar is local \LaTeX\ compilation.
For this it invokes Tectonic \cite{tectonic}, a lightweight \LaTeX\ compiler to compile the project.
This program is wrapped behind a simple API that the sidecar's frontend can call, returning the compilation status and any error messages as strings.
The PDF generated is then served back to the frontend, allowing for a live preview.
I think that the main Tectonic binary is bundled alongside the sidecar itself as part of the frontend application, though maybe it downloads it, not sure.
Regardless, upon local compilation in this way, they allow for fast iteration cycles without the need for external typesetting services.

\subsubsection{Agentic AI chat integration}
Integrating CopilotKit in the frontend allows for an agentic chat UI to come alive there \cite{copilotkit}.
For these the sidecar has the tools for the agent to fairly thoroughly manipulate documents: reading, writing, deleting and renaming files, listing the project contents, compiling \LaTeX\ and forwarding the errors if there are any, and images attached therein.
The file operations all do some path validation to ensure no traversal happens to hack out of the project folder and ensure the changes made are contained to the directory of files pertaining to the project.
With this toolset, the agent can change files and update the document accordingly if it needs to in order to fulfill the user's request.

\subsubsection{Image processing}
There are two ways that the agent may need to handle images.
Firstly, based on how the user requests this, it can decide that the image must be included in the document.
For example, it may be an image the user needs to index.
Alternatively, it may get direct user query on it.
In that case it will process it before answering a question or perhaps making a decision.
In the former scenario, the agent need not process the image, and would just need a way to copy it into the project directory.
Then it could emit code to render the image in the document.
In the latter, it would not be copied / stored, but processed by the agent.
The sidecar supplies tools for both purposes.

\subsection{Agent}
The Agent is the cloud AI behind Spartan Write.
It's responsible for interpreting the user actions, and orchestrating intelligent edits of the document.
It is built on Python FastAPI \cite{fastapi}.
Behaviour of the agent is defined in LangGraph, a framework to build stateful, multi-step AI workflows that can maintain context of the conversation and carry out complex editing logic in a deterministic way \cite{langgraph}.
It validates all user actions with WorkOS authentication \cite{workos} and streams the response back to the frontend.
Instead of being locked into one provider the agent calls into OpenRouter.
OpenRouter is a free configurable “LLM aggregation layer to choose from many language models and providers” \cite{openrouter}.
We report our usage of LLMs to PostHog \cite{posthog}.

\subsubsection{Cloud deployment}
Our agent is containerised and deployed to a virtual machine on the cloud.
Stateless means we don't hold persistent state and it means we can out scale in duplicate that agent on multiple servers.

\subsubsection{Analytics \& privacy}
Every interaction with the agent is logged and sent to PostHog \cite{posthog} for usage analytics, billing, and performance insight.
Some context of the conversation is saved for multi-turn consistency, but to protect your privacy we do not persist chats across chats.

\subsection{Benchmark}
Our benchmark tests the agent's proficiency at document editing.
We run our test dataset against the agent API, and then run a variety of scoring metrics against it.
Lending us understanding of out agent's strengths and weaknesses and helping us ultimately choose the right language model - and how performance moves as we retrain and release new ones.

\subsubsection{Testing workflow}
Our benchmark is for a simple list of document states, around a set of user prompts.
Each is a core editing task - and so we can assess the agent's proficiency across realistic use cases.
How do we run the test?
We run our benchmark against the agent server directly, via its REST API.
When benchmarking we send it a test prompt and log the response and what tools it called, and log that response in a database.

\subsubsection{Reproducibility}
If we have the test dataset and the deterministic logic we run against it to score how we benchmarked, we can just run it again against a new version of the model.
It is essential we be reproducible - any bias we will avert, and the informed decision about which language model to select.
Equally, the standardised approach allows us to track improvements - or regressions - as we alter that agent/base model.

\subsubsection{Visualisation}
"We visualise our results from the benchmarker with Plotly \cite{plotly} on a Streamlit based public web dashboard \cite{streamlit}."
The dashboard abstracts a presentation to show evaluation results with various aggregations.
Later inside of this exhibition of Playgound notes we discuss results aggregations.
This interface helps to spot both mixture of performance change, and spotting what pattern or fault models have.

\subsection{User guide}
We wrote the UserGuide atop a lightweight documentation framework called MkDocs.
It lends to generation a clean searchable website from Markdown files.
The User guide is the home of your user facing docs, with included search, mobile-responsive design, and nice clear navigation.
The user guide takes you all the way from the platform specific installation instructions for macOS, Windows and Linux, honed down until you are to quick-start tutorials for "your first text to write with Spartan Write" onward and getting to follow ons and comprehensive reference material to manage projects, configure the API, tips to get the most just and best efficient use out of your agent and best practices and good examples for troubleshooting.
